Maximo Inspection Form: The Technical Guide

Article by Ansary MARECAR published on 12 November 2025 at Dubai, United Arab Emirates.

 

The Maximo Inspection Form tool is very useful, but it has some known problems:

         The Form Designer is slow and has bugs (it's heavy, buggy, and slow).

         The system is slow on mobile devices (it has performance latency).

 

To use this tool well, you must not just look at the screen. An administrator needs to understand how the Maximo system works inside (the internal mechanisms).

 

This guide is for people who manage the system. It will give you a deep technical analysis to help you:

         Fix problems (troubleshooting).

         Make it faster (optimizing performance).

         Create better, adaptable solutions.

Phase 1: Database Architecture and Revision Control

The module's efficacy relies on a database structure that must be understood to bypass GUI limitations and synchronization issues.

A. The Data Structure of INSPECTIONFORM

The form is defined by a table hierarchy, essential for bulk management:

Maximo Table

Information

Key Technical Details

InspectionForm

Inspection form template and revision

On this table we will find: Name of the inspection, description, the current status, which revision it is. 

For each revision, a new row will be created with the original form number (origformnum) and a new incremental revision number.

You can also add a reason when you are doing any new revision.

 

Each inspection has a reference ID (inspformnum) and a type:

-          AUD Audit

-          CA Condition Assessment

-          INS Inspection

-          INV Investigation

InspectionFormStatus

Status History

The status history of the inspection form. 

 

Different type of status:

-          ACTIVE Active

-          DRAFT Draft

-          INACTIVE Inactive

-          PNDREV Pending revision

-          REVISED Revised

InspQuestion

Question and the group of questions

Each inspection has 0 to many questions. 

The question name is on description attribute.

Questions can be grouped. The group name will be saved into the same table.

The question is ordered based on groupseq number. 

If the question has a group: groupid means the group sequence 

If the question doesnt have a group: groupid will be at null.

 

The row with a group description will have: 

- a groupseq with three 0 decimals.

 

Warning: We are answering (called result) not to a question, but to a field. 

InspField

Field of the question

Each question has 0 to many fields. 

It's on these field that the user will answer (the result). 

In that way, a field have a fieldtype, a description, if it is required and visible.  

You can also choose a meter reading (attributes metername and metertype will be filled automatically depending on your meter). 

 

Different fieldtype (not all of them are available in the GUI):

-          DO Date only

-          DT Date and time

-          FU File upload

-          MM Meter readings

-          MO Multiple Choice

-          MOD Multiple choice domain

-          NR Numeric Range

-          SE Single numeric entry

-          SIG Signature

-          SO Single choice

-          SOD Single choice domain

-          TO Time only

-          TR Text response

InspFieldOption

Option of a specific field

If you have a single choice (SO) or a multiple choice (MO), the different value will be noted in this table

The attribute sequence means the order.

Each row can have an icon and a color. 

The attribute Requireaction is a boolean in the GUI. 

If is active (value at 1), it will be used for having specific rules (e.g. creating fallow up WO, SR, etc.)

InspCascadeOption

Condition-based action for option

It allows to have a condition-based action based on field response. 

Based on the answer, you can have another field visible or required.

The table can allow much more functionality in the future. 

Most of the column are not used by the GUI. 

 

B. The Data Structure of INSPECTIONRESULT

The form is defined by a table hierarchy, essential for bulk management:

Maximo Table

Information

Key Technical Details

InspectionResult

Inspection result based on template

The inspection result related to a WO/Location/Asset. 

An inspection can be achieved by batch, in a specific order. 

For instance, you WO have many assets, in which order you want to fill all these inspections results. 

You can also attach the inspection to multi-asset or task of the work order. 

The attribute referenceobject can be: 

-          WORKORDER inspection result attached to a WO level

-          ASSET inspection result attached directly to an asset

-          PARENTWO attached to launch the batch inspection

-          WOACTIVITY inspection to a WO task level

-          Etc.

 

The attribute referenceobjectid will have the wonum or ticketid. 

The createddate is corresponding to when the technician will click on Start inspection. 

Duedate is corresponding to the scheduled finished date. 

InspResultStatus

Status History of inspection result

The history of status of your inspection Result. 

 

An inspection result can have: 

-          PENDING by default at the creation

-          INPROG when you will start

-          CAN when you cancel

-          COMPLETED when you will finish

-          PENDING not proposed in the GUI by default

 

inspFieldResult

Field result

For each field, you will find here the answer. 

If you have any meter Reading, the attribute metername will be the meter name.

 

The response will be in one of these fallowing attributes:

-          Timeresponse (DO, DT, TR)

-          txtresponse (SO, TR, MM)

-          numresponse (SE)

 

Each result will have a user-id (enteredby), geolocation (plussgeojson) and a datetime (enterdate). 

 

Warning:

-          It's the datetime and the geolocation of when you open the related inspection result. 

-          It's not the datetime of when the data is transferred to the server. 

-          If the response is filled from the app, the attribute anywhererefid will be filled.

InspFieldResultSelection

Multiple selection result

If you have any meter Reading, the attribute metername will be the meter name.

The value available on this table is applicable only for multiple choice (MO) = multiple answer.

The response will be on the saved-on attribute txtresponse. 

If the response is filled from the app, the attribute anywhererefid will be filled.

C. Critical Revision Management

The record that holds the inspection data, the INSPECTIONRESULT is created at a key moment. This action locks in the specific version of the inspection form (the "form revision") that the technician must use.

         Status Control: When the Result Record is Created

o   The system uses a specific setting, called the system property (mxe.app.workorder.StatusToCreateInspection), to control when the INSPECTIONRESULT is made.

  This setting lists the Work Order (WO) statuses (like APPR - Approved, or INPRG - In Progress) that will automatically generate the result record.

  The Best Strategy: Setting this property to a status that occurs as late as possible in the process (for example, INPRG - In Progress) makes sure the technician gets the latest activated version of the form.

         Form Version Rule (The Lock)

o   The form's version is locked as soon as the INSPECTIONRESULT is created.

o   If a new version of the form is activated after the result record is made but before the technician starts entering data, the technician will still be forced to use the older, previously locked version.

Phase 2: Performance and Mobile Constraints

Module performance is directly linked to form complexity and efficient mobile synchronization.

A. Latency and Mobile Synchronization

Form loading time is a major friction point (often 8 to 30 seconds) directly related to the form's size and complexity.

Access Scenario

Behaviour

Impact

Native App (1st opening)

Very slow (download and indexing).

The structure must be downloaded and indexed into the local database ("DB Lite").

Native App (Subsequent openings)

Faster (pre-loaded data).

Reads the structure from the local database, reducing network latency.

 

Warning:

If the active inspection form is not correctly synchronized to the device, the technician will often see a blank screen in the native application because it cannot locally reconstruct the interface.

This means that if the preloaded database is updated every day at 6:00 AM, but you make a new inspection form revision at 8:00 AM, the technician who tries to fill an inspection result from the Maximo Mobile application at 2:00 PM will have a blank screen.

 

B. Best Practice: Separation by Asset Type (Granularity)

To guarantee mobile performance and data integrity for reporting, the best approach is to use distinct forms for each asset type or specialized task, especially if the team is performing that task continuously.

         Goal: Avoid a single, overly large "Mega-Form" that slows down mobile loading.

         Benefit: The technician only opens a smaller, faster form containing only relevant questions, optimizing work time and accuracy.

         Reporting Value: Separation ensures the inspection history (INSPECTIONRESULT and INSPRESULTLINE) contains only data specific to that asset type, simplifying trend analysis and Condition Monitoring.

 

However, becoming too granular will increase the time needed to fill all the inspections, and that is not the purpose of any application designed for filling inspections.

 

So, how do you find the balance?

My advice is to think about how the work is achieved by your team. Consider if they are performing an action continuously, and if your Work Order (WO) has multiple workers assigned to different tasks.

For instance, if you have Labor A doing a specific task (Task 1) on multiple assets, and Labor B doing another task (Task 2) on other assets, but both are working continuously.

 

Creating multiple individual inspections in this case would take too much time to open, close, and search.

It is sufficient to have just two inspections in this scenario:

         Task 1

         Task 2

This approach will reduce the number of individual inspections, which directly reduces the number of times the device needs to load new content.

But if your asset is costly, you can make the decision to have a longer journey to fill all the inspections per asset and the feature of batch inspection will help to avoid having this case.

 

Warning:

The batch inspection functionality will not drastically reduce the time lost when moving between individual inspections.

Furthermore, using batch inspection within a full Work Order will not help to dispatch or split the inspection tasks between two different teams working on the same Work Order.

 

Example Scenario and Limitation:

If I have a Work Order for the maintenance of all the air conditioners in a roomlet's say that room has 12 air conditionersand the first 6 units are to be maintained by Labor A while the last 6 are maintained by Labor B.

In this situation, there is nothing in the Job Plan or Route structure that would allow us to automatically create multiple inspection batches to accommodate this split in labor assignment.

B. Adaptive Planning via Conditional Tasks

Using Conditions on Job Plan Tasks (JOBPLANTASK) is the most effective way to manage diverse inspection requirements without creating numerous master Job Plans.

1.    Mechanism: A Condition is attached to a Job Plan Task.

2.    Trigger: When the Work Order is generated, Maximo evaluates the Condition (e.g., ASSETTYPE = 'MOTOR').

○     If the Condition is TRUE, the Task (and its associated inspection form) is created in the WO.

○     If the Condition is FALSE, the Task is skipped.

3.    Result: A single master Job Plan can generate different, optimized Work Orders containing only the necessary inspections. This improves technician focus and overall, Maximo performance.
